# 输出规范

## 零、docx 底层实现标准

凡是输出 `.docx` 批注版、修订版、修订批注版，修订和批注都写死为直接修改 DOCX 底层 XML 的方式实现，不采用人工打开 Word/WPS 后点击界面的方式，也不允许切换到任何非 XML 实现。

标准实现步骤：

1. 先复制原文件，严禁覆盖原件。
2. 如原合同、指令文件或工作目录位于中文路径，或路径编码表现不稳定，优先使用 PowerShell 按通配符复制到英文临时名。
3. 以共享读方式读取原 `.docx`，避免原文件被 Word/WPS 占用时无法处理。
4. 将临时英文名 `.docx` 按 ZIP 包处理，只修改必要的 XML 与关系文件。
5. 写回输出目录中的新文件，不污染原文件。

禁止退化为以下做法：

- 要求人手打开 Word/WPS 再点“修订”
- 仅在 txt 报告中描述修改意见，但 docx 中没有真实批注或修订痕迹
- 先向用户输出 md 或 txt 阶段稿，再等待确认后才生成 docx 逐条批注修订版
- 直接整篇重写 `document.xml`，导致样式、编号、分页或结构被破坏
- 直接把中文绝对路径喂给 Python 处理 docx、解包 ZIP 或改写 XML

兼容性底线：

- 输出结果应能被常见 Word/WPS 客户端直接打开
- 不依赖当前电脑是否安装 Office
- 不依赖当前运行模型是否“知道如何点界面”

对外脚本入口：

- `scripts/run_generate_review_docx.ps1`
- 示例指令：`references/review-instructions.example.json`

其中 `scripts/run_generate_review_docx.ps1` 是对外唯一入口；其内部会调用 `scripts/internal_generate_review_docx.ps1` 完成 XML 直改。`scripts/internal_generate_review_docx.ps1` 在检测到源文件、指令文件或输出路径含非 ASCII 字符时，应自动调用 `scripts/internal_stage_review_inputs.ps1`，先在英文临时路径完成 staging，再执行 XML 直改和最终回拷。

执行规则：

- 不做单独环境检测，直接运行 `scripts/run_generate_review_docx.ps1`
- 脚本运行时优先直接尝试 `.NET ZipArchive`
- 若 `.NET ZipArchive` 运行失败，则自动回退到 `Shell.Application`
- 若两种方式都失败，则直接报错，不退化为人工打开 Word/WPS 点击界面

中文路径预处理由 `scripts/run_generate_review_docx.ps1` 自动完成，用户不需要直接调用内部 staging 脚本。

推荐调用方式：

```powershell
powershell.exe -ExecutionPolicy Bypass -File .\scripts\run_generate_review_docx.ps1 `
  -Source "原合同.docx" `
  -Instructions "review-instructions.json" `
  -Output "原合同-修订批注版.docx" `
  -CleanOutput "原合同-清洁版.docx"
```

指令文件最小结构：

```json
{
  "author": "用户指定的修订人名称（未提供时可省略并默认使用“合同审核AI”）",
  "operations": [
    {
      "mode": "revision_comment",
      "anchor_text": "原条款全文",
      "replacement_text": "修订后的条款全文",
      "comment": {
        "问题": "问题说明",
        "风险": "风险说明",
        "修改建议": "修改建议",
        "建议条款": "建议条款"
      }
    }
  ]
}
```

其中 `author` 优先使用用户指定的修订人显示名称；如用户尚未提供，可省略该字段并默认使用 `合同审核AI`，不因作者名称缺失阻塞 `docx` 生成。

## 一、默认交付节奏

默认交付规则如下：

- 直接输出可打开的 `docx` 逐条批注修订版
- 不先向用户交付 `md`、`txt`、问题清单或摘要作为阶段成果
- 如需生成 JSON、txt、md、锚点表、操作清单等结构化中间材料，只能作为内部过程，生成后立即用于落版，不作为用户确认稿
- 不先生成审查报告再生成修订批注版；修订批注版是默认首个交付物
- 如用户同时要求 `docx` 与 `审查报告.txt`，应先或同时交付 `docx`；`txt` 只能作为附加产物，不得阻塞 `docx` 落版
- 凡命中合同审核流程，且用户未明确要求“先不要生成 docx”或“先只看文字意见”，默认立即进入 `docx` 落版

## 二、默认输出文件

默认生成以下文件：

- `合同名称-修订批注版.docx`

如用户明确要求，可额外生成：

- `合同名称-清洁版.docx`
- `合同名称-审查报告.txt`

如果用户只要求其中一种模式，则只输出对应文件。

## 三、默认输出目录

在原合同所在文件夹下新建：

- `合同名称-Output`

所有输出文件默认放入该目录。

## 四、批注固定结构

每条批注优先使用以下结构：

批注正文、风险说明、修改建议、建议条款默认全部使用简体中文，不得输出繁体中文。
写入 `word/comments.xml` 的批注正文必须强制锁定为 `宋体`，对应 OOXML 字体名 `SimSun`，至少写入 `w:rFonts` 的 `w:ascii`、`w:hAnsi`、`w:eastAsia`、`w:cs`。

### 问题

说明当前条款具体有什么问题。

### 风险

说明为什么这是风险，会导致什么后果。

### 修改建议

说明应如何修改。

### 建议条款

给出可直接替换或直接增补的准确条款文本。

### 修改依据（可选）

- 依据模板
- 依据法律逻辑
- 依据交易结构

## 五、批注版 OOXML 实现要求

批注版必须至少完成以下修改：

- 修改 `word/document.xml`
- 在目标文本范围插入 `w:commentRangeStart`
- 在目标文本范围结束处插入 `w:commentRangeEnd`
- 在显示批注引用的位置插入 `w:commentReference`
- 新增或更新 `word/comments.xml`
- 更新 `word/_rels/document.xml.rels`
- 更新 `[Content_Types].xml`

实现要求：

- 每条批注都必须有稳定且唯一的 comment id
- 批注正文必须写入真实 comment 节点，不得只把批注内容塞进正文段落冒充批注
- 批注正文的 `w:rPr` 必须显式写入宋体字体设置，至少包含 `w:rFonts w:eastAsia=\"SimSun\"`
- 如原文档没有 `comments.xml`，必须自动创建
- 如原文档已有批注，新增批注时必须避免 id 冲突

## 六、修订版 OOXML 实现要求

修订版必须至少完成以下修改：

- 修改 `word/document.xml`
- 将原文本包装为 `w:del`
- 将新文本包装为 `w:ins`
- 为每条修订写入 `w:id`
- 为每条修订写入 `w:author`
- 为每条修订写入 `w:date`
- 修改 `word/settings.xml`
- 在 `word/settings.xml` 中加入 `w:trackRevisions`

实现要求：

- 删除内容必须真实写入 `w:del`，不能只把旧文本直接删掉
- 新增内容必须真实写入 `w:ins`，不能只保留替换后的最终文本
- 修订应坚持最小必要改动；能改词句就不整段删除，能局部补强就不整段重写
- diff 粒度应优先按数字块、英文词块对齐，避免将连续数字或单词拆成字符级修订
- 如原文档没有 `word/settings.xml` 或缺少 `w:trackRevisions`，必须自动补齐
- 日期应写为可被 Office 识别的时间格式

## 七、修订批注版要求

若输出为 `修订批注版.docx`，默认同时满足以下条件：

- 文本修改以 `w:del` / `w:ins` 体现
- 风险说明与建议条款以 comment 体现
- 同一处问题既可有修订，也可有批注
- 除非用户明确要求，否则不得只保留其中一种

## 八、txt 审查报告模板

`txt` 审查报告不是默认交付物，仅在用户明确要求时额外生成。

如同一任务同时要求 `docx` 与 `txt`，`txt` 不得替代 `docx` 成为首个对外交付物。

建议按以下结构输出：

### 一、合同概况

- 合同名称
- 合同类型
- 审核立场
- 输入材料
- 输出模式

### 二、修改意见表

修改意见必须以表格形式逐条列出，列表头固定为：

| 原条款 | 存在风险 | 修改建议 |
| --- | --- | --- |
| 原条款全文或关键摘录 | 说明该条款的具体风险、可能后果，必要时可在单元格内补充条款定位、风险等级、问题类型 | 必须写明可直接替换或直接增补的具体条款内容，不得只写“建议完善”“建议明确”等空泛表述 |

表格输出规则：

- 一行对应一个问题点；同一条款存在多个独立问题时可拆成多行
- `原条款` 应尽量摘录原文关键内容，避免只写“第X条”
- `存在风险` 应说明风险成因和可能后果，必要时可在单元格内补充条款定位、风险等级、问题类型
- `修改建议` 必须包含完整、可落地的具体条款文本；如需分点，可在单元格内分行书写

### 三、格式及一致性问题

列出：

- 定义词
- 标点
- 编号
- 页码
- 援引
- 字体字号行距

### 四、建议进一步确认事项

列出需要业务、法务或律师进一步确认的内容。

## 九、文件命名建议

推荐命名方式：

- `合同名称-修订批注版.docx`
- `合同名称-清洁版.docx`
- `合同名称-审查报告.txt`

默认应优先落地：

- `合同名称-修订批注版.docx`

## 十、使用模式说明

### 修订模式

适合展示本轮具体修改痕迹。

### 批注模式

适合展示问题、风险和建议条款。

### 清洁版

适合定稿或推进项目时使用。

## 十一、输出后自检

生成文件后，至少检查以下事项：

- `.docx` 可以正常打开，未损坏
- 批注版存在可见批注标记或批注气泡
- 修订版存在可见插入和删除痕迹
- 批注内容结构完整，不是空批注
- 批注字体已锁定为宋体（SimSun）
- 修订的 `author`、`date`、`id` 已写入
- 原文档主要样式、编号、段落结构未被异常破坏
- 未将 `md`、`txt` 或问题清单作为阶段成果发给用户等待确认
